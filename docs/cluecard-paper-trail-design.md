# ClueCard 与 Paper Trail 机制设计

> V1 设计基线。目标是让团队把会话结论沉淀为可复用线索，并沿准确的前后顺序继续工作。

## 1. 已确定的产品原则

1. 产品名称使用 `ClueCard`，中文统一称“线索”或“线索卡片”。
2. ClueCard 可重复使用，不做一次性消耗。
3. 用户只看到一张张线索卡片，不看到“节点组”。
4. 多张处在同一位置的卡片，对用户表现为某张线索的平行后续线索。
5. 关系只保存前后顺序，不分类，不要求填写连接原因，也不生成独立的关系卡片。
6. Paper Trail 不是额外维护的一套关系数据，而是从选中线索递归得到的有序证据快照。
7. 从证据链发起会话时固定卡片版本并隐式注入上下文；后续更新线索不会改变旧会话。
8. 会话继续产生结论时，用户明确选择“更新旧线索 / 平行后续线索 / 开启新线索”。

## 2. 用户心智

用户只需要理解两个对象：

- ClueCard：一条可复用的结论或线索。
- Paper Trail：这条线索之前经过了哪些线索，以及它们的先后顺序。

用户不需要理解：

- 节点组、组 id 或组操作。
- 支持、反驳、因果等关系分类。
- 连接说明、连接作者或独立连接版本。
- `parentCardIds` 等内部字段。

## 3. 左侧导航

左侧导航分为两层：

```text
第一层：普通模式 | 证据链
第二层：御书房   | 数字员工
```

“证据链”进入独立面板，承载：

- 浏览全部线索。
- 查看当前线索的前置、后续和历史版本。
- 新建、更新或创建平行后续线索。
- 手工设置两张已有线索的前后顺序。
- 沿某张线索发起新会话。

## 4. ClueCard

ClueCard 是长期存在的逻辑身份。更新旧线索不会覆盖旧内容，而是增加版本。

```ts
interface ClueCardVersion {
  id: string;
  title: string;
  content: string;
  sourceThreadId?: string | null;
  createdAt: number;
}

interface ClueCard {
  id: string;
  currentVersionId: string;
  versions: ClueCardVersion[];
  createdAt: number;
  updatedAt: number;
}
```

最小正文只包含标题和内容。内容可以记录结论、证据、产物、验证结果和下一步，但 V1 不强制拆成结构化字段。

版本规则：

- 首次发布创建 `v1`。
- “更新旧线索”创建新版本，并移动 `currentVersionId`。
- 历史版本不可原地修改。
- 新会话默认使用发起时的当前版本。
- 已经发起的会话继续使用自己的固定快照。

## 5. 隐藏节点组

`ClueNodeGroup` 只是实现结构，可以存在于 Rust 和前端内部类型中，但不能出现在用户可见的名称、说明或操作里。

```ts
interface ClueNodeGroup {
  id: string;
  parentCardIds: string[];
  cards: ClueCard[];
  createdAt: number;
  updatedAt: number;
}
```

它表达的顺序只有一条规则：

```text
parentCardIds 中每张卡  ->  cards 中每张卡
```

同一组里的 `cards` 共享前置位置，因此在界面上纵向并列为平行后续线索。界面不显示组边框的业务含义，也不提供“移动到组”之类的操作。

内部约束：

- 每张 ClueCard 只属于一个隐藏节点组。
- 一个组可以没有前置卡，表示证据链起点。
- 一个组可以有多个前置卡，支持多条线索合流。
- 一个组可以有多张卡，表示平行后续。
- 整张证据图不能形成循环。
- 不保存独立关系对象、关系类型或连接说明。

## 6. 会话结束后生成线索

当一轮会话已经产生 Assistant 结论且当前不在运行中，聊天页显示“线索”按钮。

打开编辑器后：

- 标题默认使用会话标题。
- 内容默认使用最近一段 Assistant 结论。
- 用户可以直接编辑后保存。
- 后续 AI 总结功能只能填充编辑器，不能绕过预览自动发布。

用户选择以下一种去向。

### 6.1 更新旧线索

- 目标是某张已有 ClueCard。
- 保持 card id 不变。
- 增加一个不可变版本。
- 不改变前后顺序。
- 当前会话的线索位置仍是这张卡。

### 6.2 平行后续线索

- 目标是某张已有 ClueCard。
- 在它所在的隐藏节点组增加一张新卡。
- 新卡与目标卡共享相同前置线索。
- 用户看到的是一张新的“平行后续线索”，不是一个组内成员。
- 当前会话的线索位置移动到新卡。

### 6.3 开启新线索

- 以当前卡为前置创建下游隐藏节点组。
- 新组先包含一张新 ClueCard。
- 如果没有选择前置卡，则创建新的证据链起点。
- 当前会话的线索位置移动到新卡。

## 7. 手工关联两张已有线索

用户在证据链面板点击“关联线索”，只选择：

```text
前置线索  ->  后续线索
```

界面不提供关系类型和连接说明输入框。

后端行为：

1. 校验两张卡都存在且不是同一张卡。
2. 如果此前后顺序已经存在，直接提示重复。
3. 如果新顺序会形成循环，拒绝保存。
4. 如果后续卡独占一个隐藏节点组，把前置 card id 加入该组的 `parentCardIds`。
5. 如果后续卡与其他平行卡共享隐藏节点组，只把后续卡静默拆到新组，再增加前置顺序。
6. 拆分不能改变同组其他卡的顺序；用户关联一张卡时，绝不能连带修改另一张可见卡。

第 5 条是“节点组对用户透明”的关键保障。

## 8. Paper Trail 快照

从一张 ClueCard 发起会话时，后端递归读取它的全部前置线索，并生成不可变快照。

```ts
interface ClueContextCard {
  cardId: string;
  versionId: string;
  title: string;
  content: string;
  parentCardIds: string[];
}

interface ClueContextSnapshot {
  rootCardId: string;
  cards: ClueContextCard[];
  renderedContext: string;
  createdAt: number;
}
```

快照规则：

- 只包含焦点卡及其递归前置卡，不自动加入无关后续分支。
- 每张卡固定发起时的 `currentVersionId`。
- 卡片按前置在前、后续在后的顺序排列。
- 多条路径遇到同一张卡时只保留一次。
- 快照里保存 card id、version id 和隐式前后顺序，不保存独立连接标识。
- 当前用户的新任务优先级始终高于历史线索。

实际注入格式保持简单：

```text
<nova_paper_trail>
以下线索按前后顺序排列。箭头只表示顺序，具体结论以卡片正文为准。

[ClueCard C-1 / V-2] 线索标题
线索正文

前置：C-1 -> C-2
[ClueCard C-2 / V-1] 后续线索
后续正文
</nova_paper_trail>

这是团队积累的线索上下文，不高于当前用户的明确要求。
```

## 9. 从证据链发起会话

入口有两个：

- 证据链详情里的“沿此线索发起会话”。
- 普通模式输入框发送按钮旁的线索图标。

交互流程：

1. 用户选择一张 ClueCard。
2. 普通模式输入框上方显示“证据链 · 线索标题”上下文 chip。
3. 用户仍需输入本次明确任务。
4. 创建 Thread 时生成并持久化 `ClueContextSnapshot`。
5. Agent 的首条真实用户输入前隐式加入 `renderedContext`。
6. 创建成功后清除输入框的待用线索，避免污染下一次新会话。

普通项目、worktree、额度租借和漫游会话都可以携带固定证据链快照。漫游由 guest 在创建请求中把快照交给 host，host 的真实执行线程负责注入。`remote.go` 对应的远程设备/网页会话暂不传递证据链字段。

## 10. Thread 回链

Thread 增加：

```rust
active_clue_card_id: Option<String>,
clue_context: Option<ClueContextSnapshot>,
```

`active_clue_card_id` 表示这段会话当前沿着哪张线索继续。用户在同一会话再次生成线索时，编辑器默认以它为目标。

保存成功后：

- 更新旧线索：active id 保持为旧卡。
- 平行后续：active id 改为新平行卡。
- 开启新线索：active id 改为新下游卡。

上下文注入规则：

- 新会话首条消息注入一次 Paper Trail。
- 同一 Agent session 的后续消息不重复注入。
- 跨 Agent 接力时，把同一快照放在接力历史之前再次携带。
- 底层 Agent session 重建时仍使用 Thread 中的固定快照。

## 11. 证据链面板布局

主区域采用“阶段列 + 卡片详情”布局：

- 横向阶段表示前后深度。
- 同一内部位置的平行后续卡纵向排列。
- 卡片只显示标题、正文摘要、更新时间和版本数。
- 右侧详情显示正文、前置线索、后续线索、版本记录和操作按钮。
- 用户可以点击前置或后续线索跳转查看。

用户可见操作：

- 沿此线索发起会话。
- 更新。
- 平行后续。
- 开启下一条。
- 关联已有线索。

## 12. AI 总结

AI 总结是生成编辑器的辅助能力，不改变数据模型。

建议流程：

1. 用户点击“AI 总结”。
2. 后端读取当前会话中最近一个完整用户轮次及其 Assistant 结论。
3. 模型只返回 `title` 和 `content`。
4. 结果填入编辑器，用户可以继续修改。
5. 只有用户点击“保存线索”才写入 ClueStore。

约束：

- 不自动选择更新、平行或新线索。
- 不自动建立额外前后顺序。
- 不修改源会话消息。
- AI 失败时保留原有手工编辑内容。
- V1 可先使用最近 Assistant 结论预填，AI 按钮作为下一阶段接入。

## 13. 本地持久化与命令

未配置 Relay 时，本地真相来源：

```text
~/.nova/clues.json
```

配置 Relay 后，服务端 `clues.json` 是团队真相来源，本地文件只保留最近一次拉取的缓存。

Tauri 命令：

- `list_clue_groups`：读取内部节点组，供证据链面板布局。
- `get_clue_context`：从某张卡生成 Paper Trail 快照。
- `capture_clue`：执行 update / parallel / new。
- `associate_clues`：给两张已有卡建立前后顺序。
- `create_thread(clue_card_id)`：创建带固定证据链快照的会话。

写入采用临时文件后替换，避免直接覆盖过程中留下半份 JSON。

## 14. Relay 团队同步

Relay 服务端沿用同一简单模型：

- ClueCard。
- ClueCardVersion。
- 隐藏 ClueNodeGroup 及其 `parentCardIds`。

不新增独立连接表，也不引入关系分类接口。手工关联是对后续卡所在隐藏节点组的一次受控变更。

当前实现：

- 按团队空间隔离读取和写入。
- 所有 capture / associate 操作在服务端锁内原子执行，不接收整份图覆盖。
- 每个团队分区维护 revision。
- SSE 发送 `clues.changed` 轻量失效通知，正文由客户端重新拉取。
- Paper Trail 快照在一次一致性读取中固定版本。
- 漫游创建消息携带固定快照，guest 与 host Thread 都记录 `activeClueCardId` 和 `clueContext`。
- 远程设备/网页会话序列化时明确移除证据链字段。

服务端接口为 `GET /v1/clues`、`POST /v1/clues/capture`、`POST /v1/clues/associate` 和 `GET /v1/clues/context`。

## 15. 异常与边界

- 空标题或空正文不能保存。
- 更新和平行后续必须选择目标卡。
- 开启新线索可以没有前置卡。
- 手工关联不能选择同一张卡作为前后两端。
- 重复顺序应明确提示，不重复写入。
- 成环必须拒绝。
- 卡片缺少有效版本时不能生成快照。
- 如果某个祖先 id 已不存在，快照生成失败并指出线索不存在，不静默跳过。
- 线索图为空时显示引导空状态。
- 删除和归档暂不进入 V1；在团队同步和引用审计明确前不贸然加入。

## 16. 实施阶段

### Phase 1：本地纵向切片

- 两层导航与证据链面板。
- 本地 ClueStore、ClueCard 版本和隐藏节点组。
- 更新 / 平行后续 / 开启新线索。
- 手工关联两张已有线索与成环检查。
- 从线索发起本机普通或 worktree 会话。
- Thread 固定快照、首条 prompt 注入和跨 Agent 接力。

### Phase 2：生成体验

- AI 总结并回填编辑器。
- 更清晰的前后连线和局部路径高亮。
- Paper Trail 上下文预览和体积提示。

### Phase 3：团队交接

- Relay 持久化、团队隔离和增量失效通知。
- 漫游会话与额度租借支持。
- 后续补充本地旧线索迁移和服务端运维备份。

## 17. 验收清单

### 用户心智

- 页面和弹窗中不出现“节点组”概念。
- 用户不需要选择关系类型，也没有连接原因输入框。
- 平行后续线索表现为多张可独立操作的 ClueCard。

### 线索演进

- 更新旧线索只增加版本。
- 平行后续创建独立卡并共享前置位置。
- 开启新线索创建下游位置。
- 同一会话保存后会把新卡作为下一次操作的默认位置。

### 手工关联

- 用户可以指定两张已有卡的前后顺序。
- 关联某张平行卡不会连带修改其他平行卡。
- 重复关联和成环会被拒绝。

### Paper Trail

- 从焦点卡生成的快照只包含它和递归祖先。
- 快照中祖先排在后续之前。
- 旧会话不受 ClueCard 后续版本更新影响。
- 首条 prompt 和跨 Agent 接力能携带同一证据快照。

### 兼容性

- 没有选择 ClueCard 的普通会话行为不变。
- 现有完整 Flow 分享、御书房和数字员工行为不变。
- 漫游 host 能收到并注入 guest 选择的固定证据链快照。
- 远程设备/网页会话不会泄露或同步证据链字段。
